# 文本与通用 OpenAI 学习笔记

## 先学什么

文本工作流是最好的入门点，因为它能帮助你掌握许多 OpenAI API 共通的请求结构。

核心思路：

- 为任务选择合适模型。
- 使用 Responses API 直接发起模型请求。
- 将提示词放在代码中维护，而不是依赖远端提示对象。
- 当提示行为很关键时，加入轻量评测检查。

## 关键概念

### 1. 模型选择

分类、改写、抽取这类任务优先用更小、更便宜的模型。规划、分析、多步逻辑、或更高质量输出，优先用推理能力更强的模型。

### 2. Responses API

在许多现代文本与多模态场景中，Responses API 是推荐路径。一次请求通常包含：

- `model`
- 用于高优先级行为约束的 `instructions`
- 任务内容 `input`

### 3. 提示词设计

高质量提示词通常包含：

- 清晰目标
- 约束条件
- 输出格式要求
- 当任务有歧义时提供示例

### 4. 结构化输出

当应用需要稳定 JSON 时，应定义 schema 并在下游使用前做校验。

## GPT 模型演进（偏官方信息的总结）

说明：OpenAI 模型能力与可用性会持续变化。上线前请始终核对官方模型页、发布说明与价格页。

### 阶段 A：GPT-3.5 时代（实用基线）

典型价值：

- 响应快、成本低。
- 适合轻量任务，如改写、分类、简单抽取。

典型限制：

- 多步推理较弱。
- 在复杂约束下遵循指令稳定性一般。
- 在代码与长上下文任务上质量波动更大。

### 阶段 B：GPT-4 时代（质量与推理跃升）

典型价值：

- 推理可靠性与指令遵循显著提升。
- 代码、分析、复杂写作质量更高。
- 相比早期代际，安全行为更强。

典型限制：

- 延迟与成本通常高于小模型。
- 对高并发、实时交互场景的吞吐表现可能不占优。

### 阶段 C：GPT-4 Turbo / 优化版 GPT-4（效率升级）

典型价值：

- 相比早期 GPT-4 版本，性价比更好。
- 更大的上下文窗口（适合长提示与大文档）。
- 在有成本约束的生产系统中更实用。

典型限制：

- 对最难的推理链路，仍不一定优于更新的专用推理模型。

### 阶段 D：更新的 GPT 家族（例如 GPT-5.x）

典型价值（按官方发布定位）：

- 代码与类 Agent 任务完成能力更强。
- 长上下文理解更强（适合大代码库与长文档）。
- 在许多实际工作负载中，指令遵循与输出一致性更好。

典型限制：

- 选型仍需在质量、速度、预算间权衡。
- 并非所有任务都能同等受益于更大更强模型。

### GPT-5 各版本差异（速查）

说明：以下数字基于撰写时官方文档，后续可能变化。生产上线前请再次核对模型页与价格页。

| 模型 | 定位 | 上下文窗口 | 最大输出 | 文本价格（每 1M token） | 推理强度默认值 | 常见场景 |
|---|---|---|---|---|---|---|
| GPT-5.5 (`gpt-5.5`) | 面向最复杂专业任务/代码任务的最高质量 | 1,050,000 | 128,000 | 输入 $5.00，输出 $30.00 | 支持 `none/low/medium/high/xhigh`，默认 `medium` | 高难多步推理、复杂编码、高风险 Agent 工作流 |
| GPT-5.4 (`gpt-5.4`) | 前沿质量，成本性能优于 5.5 | 1,050,000 | 128,000 | 输入 $2.50，输出 $15.00 | 支持 `none/low/medium/high/xhigh`，默认 `none` | 预算更紧时的高质量代码/分析 |
| GPT-5.4 mini (`gpt-5.4-mini`) | 高吞吐 mini 模型，实战能力强 | 400,000 | 128,000 | 输入 $0.75，输出 $4.50 | 支持 `none/low/medium/high/xhigh`，默认 `none` | 生产聊天、子代理、工具密集型大规模场景 |
| GPT-5.4 nano (`gpt-5.4-nano`) | GPT-5.4 系里最便宜，主打速度/成本 | 400,000 | 128,000 | 输入 $0.20，输出 $1.25 | 支持 `none/low/medium/high/xhigh`，默认 `none` | 分类、抽取、排序、轻量自动化 |

一些影响成本规划的附加细节：

- 在 GPT-5.5 与 GPT-5.4 的长上下文会话中，超大输入（官方文档提及 >272K token）可能触发更高会话定价倍数。
- `mini` 与 `nano` 通常是高 QPS 的首选；复杂/失败请求再升级到 `gpt-5.4` 或 `gpt-5.5`。
- 各层级的工具能力可能有差异（例如官方文档显示，`gpt-5.4 mini` 在计算机使用/工具搜索支持上通常强于 `gpt-5.4 nano`）。

### GPT-5 家族简单路由策略

1. 短小且确定性的任务（抽取/分类/排序）先用 `gpt-5.4-nano`。
2. 一般产品流量默认用 `gpt-5.4-mini`。
3. 质量不达标时升级到 `gpt-5.4`。
4. 最难推理/编码或高端档位再用 `gpt-5.5`。
5. 对难例，先提高 `reasoning.effort`，再考虑换更高模型档位。

### 并行路线：推理导向模型

在 OpenAI 的产品定位中，一些新模型通过增加推理时算力，强化推理深度。

典型价值：

- 在高难数学、逻辑、规划、代码分解任务上表现更好。
- 在复杂工作流中的多步工具调用更稳健。

典型权衡：

- 对复杂查询，延迟与成本通常高于 mini/快速模型。

## 各代模型在实战中的性能差异

与其只问“哪个更新”，更建议用下面维度比较系统表现。

| 维度 | GPT-3.5 类基线 | GPT-4 代 | 更新 GPT（如 GPT-5.x）与推理导向代 |
|---|---|---|---|
| 复杂推理 | 基础到中等 | 强 | 在高难多步任务上更强 |
| 指令遵循 | 简单提示下尚可 | 更稳定 | 受约束任务中稳定性最高 |
| 代码质量 | 适合片段 | 架构层面更好 | 仓库级与 Agent 式编码更好 |
| 长上下文 | 有限到中等 | 后期版本更大 | 在新代际中表现最佳 |
| 多模态理解 | 早期文本模型通常有限/无 | 已引入并增强 | 更成熟的多模态执行能力 |
| 延迟与成本 | 效率最好 | 质量更高但更贵 | 按层级分化（mini / full / reasoning） |
| 工具使用/函数调用 | 基础 | 更好 | 在复杂工具链更可靠 |

## 生产评测矩阵：应测什么

官方长期强调评测，而非直觉。建议构建同时包含“基准指标 + 业务指标”的评分卡。

### 1. 能力基准（公开/常见）

- 知识与推理：MMLU、GPQA 类高难问答。
- 数学与逻辑：MATH、GSM8K 类任务。
- 代码：HumanEval、MBPP、SWE-bench 类端到端修复任务。
- 多模态：MMMU 及相关图文理解基准。

### 2. 任务级产品指标（你自己的评测集）

- 正确性：精确匹配、事实一致性、schema 合法性。
- 指令遵循：格式合规、策略合规。
- 工具成功：函数调用准确率、重试次数、最终任务完成率。
- 用户价值：采纳率、与目标输出的编辑距离、人类偏好胜率。

### 3. 系统指标

- P50/P95 延迟。
- 单次成功任务成本。
- 并发下吞吐。
- 失败率与回退频率。

### 4. 安全与可靠性指标

- 有害输出拒答质量。
- 高风险提示上的幻觉率。
- 检索/工具工作流中的提示注入抵抗力。
- 在提示词/版本更新后的稳定性。

## 新模型通常增强了什么（官方高层视角）

OpenAI 不会公开每代的全部架构细节，但官方文档与 system card 通常指向这些趋势：

- 更好的后训练对齐（指令微调、偏好优化、安全调优）。
- 更强的工具使用训练（函数调用、多步任务编排）。
- 长上下文优化（训练与推理层面的长上下文检索与推理提升）。
- 多模态一体化提升（文本 + 图像 + 音频流程更实用）。
- 更强调可靠性的训练与评测闭环（更多对抗与策略测试）。
- 在推理导向模型中，对困难任务采用更审慎的推理时计算。

## 模型选型作战手册（简单且有效）

1. 从任务类型出发，而不是从模型热度出发。
2. 用真实用户输入构建 100-300 条评测样本。
3. 至少比较 1 个 mini 模型和 1 个强模型。
4. 在同一张表里同时打分质量、延迟、成本。
5. 选择“满足质量门槛的最低成本模型”。
6. 为高风险或失败场景准备回退模型。

## 最小 Python 模式

```python
from openai import OpenAI

client = OpenAI()

response = client.responses.create(
    model="gpt-5.4-mini",
    instructions="You are a concise assistant.",
    input="Summarize the advantages of retrieval-augmented generation in 4 bullet points."
)

print(response.output_text)
```

## 建议练习

1. 做一个长笔记摘要器。
2. 做一个按主题标注反馈的分类器。
3. 做一个简历、发票或工单的 JSON 抽取器。
4. 在同一任务上比较小模型和强模型。
5. 加一个评测脚本，记录每个模型的正确性、延迟、成本。

## 官方评测流程说明

- OpenAI 文档强调“迭代评测”是生产实践核心。
- 官方评测流程通常是：定义任务 -> 在代表性数据上运行 -> 分析结果 -> 迭代优化。
- 在当前官方文档中，旧版 Evals 平台正在逐步退场；新工作流建议从 Datasets 与新评测指引开始。
- 维护你自己的黄金测试集，并在每次提示词/模型升级时回归，防止退化。

## 负责任 AI 速记（AI-901 高频）

设计 AI 应用时，可用以下六项原则做实用检查清单：

1. 公平性（Fairness）
    确保输出不会系统性地不利于特定群体。
2. 可靠性与安全性（Reliability and Safety）
    覆盖正常与边界场景验证模型行为，并定义安全回退。
3. 隐私与安全（Privacy and Security）
    保护敏感数据、最小化保留、采用最小权限访问。
4. 包容性（Inclusiveness）
    面向多元用户、语言与可访问性需求进行设计。
5. 透明性（Transparency）
    明确告知何时使用 AI，并说明能力边界。
6. 责任性（Accountability）
    明确模型行为、事故处理与策略合规的责任主体。

实践中，可将这些原则映射到评测矩阵：质量指标、安全检查、延迟/成本与发布门禁。

## AI-901 模块深挖（文本与策略）

### 1. Anomaly Detector

它是什么：

- 用于识别数值时间序列中异常模式的服务。
- 支持单变量与多变量异常检测思路。

适用场景：

- 监控流量、延迟、CPU、传感器数据、财务趋势突变。
- 需要在“早预警”与“减少误报疲劳”之间平衡的场景。

生命周期提醒：

- Microsoft Learn 当前说明已无法创建新的 Anomaly Detector 资源，且服务已规划退役。备考中可作为概念考点，实际项目中可作为迁移规划议题。

工程设计要点：

- 季节性与趋势处理会显著影响误报率。
- 缺失数据与不规则采样需要预处理。
- 告警机制应包含置信阈值与抑制窗口。

### 2. Language Understanding (LUIS)

它是什么：

- 经典 NLU 服务，核心是意图识别与实体抽取。
- 关键设计对象包括 intents、entities、utterances、phrase features。

为什么重要：

- AI-901 常考对话流程里的“意图 vs 实体”区分。
- 有助于你在使用生成式对话前，先理解确定性 NLU 管线。

生命周期提醒：

- 当前 LUIS 文档多在 previous-versions/archive 语境中。新生产项目可将其视作基础概念，并评估最新 Azure Language 与 Foundry 方案。

### 3. NLP 技术选型（架构层面）

常见误区是把传统 NLP 与大语言模型任务混在一起。

当你需要这些时，优先传统 NLP：

- 结构化输出（标签、实体、情感分数、关键词）。
- 更强确定性与更易审计。
- 窄任务下更低成本。

当你需要这些时，优先生成式语言模型：

- 开放式生成、摘要、改写、复杂问答。
- 处理异构输入的丰富上下文能力。
- 带工具编排的 Agent 工作流。

能混合就混合：

- 先做 NLP 抽取，再把结构化信号喂给模型做推理或报告生成。

### 4. Azure Machine Learning (AML)

它是什么：

- 覆盖 ML 全生命周期的平台：数据准备、训练、部署、端点与 MLOps 治理。

何时选 AML 而不是预构建 AI 服务：

- 需要基于领域私有数据进行自定义训练。
- 需要严格 MLOps 管控（实验追踪、模型注册、发布策略）。
- 需要超出托管 API 默认能力的性能调优。

何时预构建 AI 服务更优：

- 需要最快上线并尽量少做基础设施运维。
- 任务与现有内置能力高度匹配。

速查矩阵：

| 需求 | 更适合预构建 AI 服务 | 更适合 Azure Machine Learning |
|---|---|---|
| 最快交付 | 是 | 否 |
| 基于私有数据做自定义训练 | 有限 | 是 |
| 深度 MLOps 与生命周期控制 | 有限 | 是 |
| 低运维负担 | 是 | 否 |
